Where Canary Lab fits

Canary Lab is one piece of a workflow that usually involves a test runner, sometimes a CI dashboard, and a coding agent. This page is a map of where it sits and what it pairs with — not a feature contest.

① What it is

A local orchestrator wrapped around a Playwright run

Canary Lab owns the lifecycle around a local E2E run: starting the services a feature depends on, applying environment files, running Playwright, and writing per-run logs and artifacts to a stable directory. When a test fails, it prepares a structured failure context that an external coding agent — Claude Code or Codex — can read and act on.

  • Starts the services a feature needs and waits for health checks before tests run
  • Swaps environment files (envsets) for the duration of a run, then restores them
  • Runs Playwright and captures stdout, events, screenshots, videos, and traces under logs/runs/<runId>/
  • Slices service logs per failed test so the relevant window is findable without scrollback
  • Optionally hands the failure context to a coding agent and watches for restart/rerun signals

② What it isn't

The boundaries

Canary Lab fills a specific gap. Several other tools cover stages it explicitly does not, and that's intentional — the value is in pairing well, not in covering everything.

  • Not the test runner. Playwright is. Canary Lab invokes it.
  • Not the coding agent. Claude Code, Codex, or another CLI is. Canary Lab supplies context and signal files.
  • Not a test healer. It doesn't rewrite selectors, waits, or assertions in test files.
  • Not a CI dashboard. It doesn't aggregate runs across a team or expose hosted history.
  • Not cloud SaaS. It runs locally on a single developer's machine.

③ The lifecycle

④ The neighbors

How each tool relates

Grouped by what they're for. None of these are competitors to Canary Lab — they cover different stages of the lifecycle, operate at different scales, or address different audiences. Most can coexist with Canary Lab on the same project.

Test runners & test healers

Playwright · Playwright Healer

Playwright is the runner Canary Lab invokes — there's no overlap, just a hand-off. Playwright Healer is a layer on top of Playwright that patches test code (selectors, waits, assertions) using DOM, console, and network signals from a replay. Canary Lab works in the opposite direction: it leaves test files alone and instead surfaces the system state behind the failure so an agent can fix the implementation. The two could coexist on the same project — Healer for selector drift, Canary Lab for app-level bugs.

Test-authoring SaaS

Octomind · Mabl · QA Wolf

These are vendor-managed services that generate and maintain tests for you, often with their own AI in the loop. They cover the authoring stage and run the tests on the vendor's hosted runner. Canary Lab assumes you already write your tests in Playwright and care about what happens locally during a failing run. Different operating model, different stage in the lifecycle.

Test history & CI dashboards

Currents.dev

Currents is a cloud service for aggregating Playwright runs across a team's CI: history, flakiness analytics, retry tracking, parallelisation. It sits on the CI side of the lifecycle. Canary Lab sits on the local-dev side. They're complementary — a team could run Currents in CI and Canary Lab on each developer's machine without conflict.

Local stack orchestrators

Tilt · Skaffold

Tilt brings up a Kubernetes-based dev stack with hot reload. It's adjacent to Canary Lab in that both "boot local services," but the runtimes and goals differ — Tilt for ongoing development against a K8s stack, Canary Lab for per-run process lifecycle around a Playwright invocation. A project using Tilt for dev could still use Canary Lab for the test loop.

Coding agents

Claude Code · Codex · Goose

These are the other half of Canary Lab's heal loop. Canary Lab doesn't write code; it produces a structured failure context (sliced logs, heal-index.md, signal files) that one of these agents can act on. Claude Code and Codex are already wired up; Goose follows the same general pattern and could be added. The complement is direct: agents are good at editing code, Canary Lab is good at handing them the right context to edit it from.

Agent envelopes at other scales

Stripe Minions

Stripe's Minions is internal-only tooling: a fork of Goose extended with "blueprints" (deterministic workflow scaffolds), 400+ MCP tool integrations, and pre-warmed devboxes, used to autonomously produce ~1,300 PRs per week. The interesting connection is that Canary Lab is the same pattern at a much smaller scale — both wrap a coding agent with scaffolding and context to make it more reliable on a constrained kind of task. Stripe's scope is "any small bounded engineering task across the company"; Canary Lab's is "fix the local app code that broke this Playwright run." Same shape, very different blast radius.

Chat clients with code access

Claude Desktop

Claude Desktop is the Anthropic chat app for Mac and Windows. With MCP, it can read files and call tools, but it's interactive-first and not designed to be driven headlessly from a runner. It's adjacent to Canary Lab — a developer might use it to ask questions about a captured run — but it isn't a substitute for Claude Code or Codex in the auto-heal slot.

⑤ A worked example

How the tools tile

A concrete picture of where Canary Lab sits in a real workflow.

Suppose you're working on a 3-service local stack — a frontend, an API, and a worker — with Playwright tests that exercise the whole thing. A test fails in the checkout flow.

  • Run the test locally. Canary Lab boots the three services, applies the right envset, runs Playwright, and captures logs, traces, and the failure under logs/runs/<runId>/.
  • See what broke. The Canary Lab UI shows per-test logs, the Playwright trace, and a sliced view of the relevant service log window. No terminal scrollback to dig through.
  • Hand it to an agent. Claude Code or Codex reads heal-index.md and the failure slice, edits the API handler, and writes a .rerun signal.
  • Track flakiness over time. If the team also runs Playwright in CI, Currents collects the cross-team history independently — Canary Lab doesn't replace that.
  • Maintain the tests themselves. If selector drift is the recurring problem, Playwright Healer or Octomind handle that — Canary Lab leaves test files alone.

Each tool has a stage. Canary Lab fills the local debug stage and the agent hand-off, which is the gap most other tools don't directly target.
